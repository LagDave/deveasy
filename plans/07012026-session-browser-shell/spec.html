Why

DevEasy today gives an agent session a terminal but no eyes. A per-session, persistent browser lets the operator log in once (session survives refresh) and watch the agent drive that same page live — extending the app from terminal use to a visual view of what the agent is working on. Testing web flows, filling forms, and reproducing UI bugs currently happen blind; this closes that gap.

What

A browser shell scoped to a DevEasy session, shown side-by-side with that session's chat. One persistent Chromium per session (login/cookies survive reload and DevEasy restart), streamed to a canvas the operator can click and type into, with tabs. The same Chromium is driven by the agent through a custom in-process MCP server exposing semantic tools (navigate / click / type / read page / tab control). Done when: the operator opens a browser in a session, logs into a site, reloads DevEasy and is still logged in; the agent navigates and interacts via MCP tools while the operator watches; a manual close frees the Chromium's memory but keeps the login; deleting the session closes the Chromium and wipes its profile; and an idle browser self-suspends.

Decisions locked (grill, 2026-07-01): Scope = per session (not project) so each agent's browser is isolated and rides alongside its chat. Layout = one session at a time; the active session's panel splits into chat + browser (the panel only ever mounts one activeSessionId). Agent control = custom MCP server. Engine = Playwright (bundled Chromium). Tool model = semantic (selectors + accessibility tree). v1 includes tabs, idle eviction, and an input-contention lock.

Context

Three-state lifecycle (the core model)

  • Close / suspend — kill the Chromium process, reclaim ~250MB RAM. Profile dir stays. Reopen re-launches against the same profile, login intact. This is the "manually closeable that frees memory" ask. Mirrors POST /api/sessions/:id/stop.
  • Delete session — close the browser and wipe the profile dir. Login gone. Hooks into SessionsController.remove next to the existing SessionStreamService.stop call.
  • Server restart / crash — process dies (in-memory registry cleared), profile dir survives on disk. Browser comes back "closed but re-openable." A startup GC reaps profile dirs with no matching session.

Patterns to follow

  • Backend service registry: mirror TerminalProcessService exactly — static Map, typed BrowserError (§8.3), signal reaping, attach returns a detach fn that never kills.
  • REST surface: mirror gbp-automation/ — thin controller → feature-services/, ok()/fail()/handleBrowserError() in feature-utils/, { success, data, error } envelope (§8.1). Input validated at the boundary (§11.2, §5.2).
  • Protocol surfaces are envelope exceptions: the WS relay and the MCP endpoint speak their own protocols, not {success,data,error} — same accepted deviation as the existing /ws/* relays. Everything under /api/browser keeps the envelope.
  • Frontend: mirror useTerminalSocket + the api/ triad (§12.1, §14.2, §14.3, §15.1). No raw fetch, data-fetching in hooks, server state via React Query.
  • Logging: childLogger({ module }), Pino only (§9.1).

Reference files: src/services/TerminalProcessService.ts (service/registry), src/controllers/gbp-automation/ (REST shape), frontend/src/hooks/useTerminalSocket.ts (client). The MCP server has no in-repo analog (zero MCP exists today) — structure it as a clean single-purpose module using @modelcontextprotocol/sdk's Streamable-HTTP transport, tools defined declaratively.

Agent-control architecture (MCP)

  • DevEasy hosts one in-process MCP server over Streamable HTTP, mounted at /mcp/browser on the existing Express server (no new port). It is co-located with BrowserProcessService, so tool calls drive the Playwright pages directly — no cross-process hop.
  • On spawn, ClaudeProcessService writes a per-session MCP config and passes --mcp-config <file>. The config points at http://127.0.0.1:<port>/mcp/browser and carries the DevEasy sessionId + a generated per-session token in headers.
  • The MCP server resolves sessionId → browser from the registry, validates the token (§5.4), lazily launches the Chromium on first tool call, and executes semantic actions. Agent tool calls appear in the session transcript automatically (they are tool_use events in the existing CLI stream), so "watch it alongside chat" needs no extra wiring.

Constraints

Must

  • Mirror TerminalProcessService for the registry/lifecycle and gbp-automation/ for the REST surface (§6.1).
  • Persist profiles per session on disk; login survives reload and DevEasy restart.
  • Human WS input and agent MCP tool calls drive the same Chromium via the one registry.
  • Enforce the input-contention lock server-side in BrowserProcessService (§5.4) — the single chokepoint both paths pass through.
  • Bind/guard the MCP endpoint: localhost only + per-session token validated on every call (§5.2, §5.4). Token generated, never hardcoded (§5.1).
  • Cap concurrent browsers and idle-evict; keep the constants in config/ (§4.2).
  • Reap Chromium process trees on SIGINT/SIGTERM/SIGBREAK (Windows-safe teardown).
  • Keep every new file within size tiers, <800 hard ceiling (§2.4/§13.1); split into feature-services//feature-utils/ if a service grows.
  • Tests for the service, MCP tool handlers, and REST controller (§20.1, §20.2).

Must Not

  • No real <iframe> to live sites (framing is blocked by X-Frame-Options/CSP) — pixel screencast only.
  • No DB tables — browsers are in-memory processes like terminals; profiles are the only durable state.
  • Do not weaken the API-key scrub in the CLI child env (subscription auth must hold).
  • Do not route /api/browser responses outside the {success,data,error} envelope.
  • Do not write profiles into the user's project dir (they are session-scoped, not project-scoped, and must not be git-injected).
  • No new frontend dependency — screencast renders to a native <canvas>/<img>.
  • No drive-by refactors of the terminal or session code (§ Refactoring Rule).

Out of scope (v2)

  • Multiple sessions' browsers tiled on screen at once (panel is one-session-at-a-time).
  • Coordinate/vision-based agent tools (semantic only in v1).
  • Browser extensions, downloads manager, devtools passthrough, per-tab profiles.
  • Moving profiles to an OS app-data dir (v1 uses a DevEasy-managed state dir under the repo root).

Risk

Level: 3 (Structural Risk) — new heavy dependency, a net-new MCP subsystem, a new WS surface, and a change to the CLI spawn path that every session flows through.

Risks identified

  • CLI support for HTTP MCP servers is version-dependent. The installed claude CLI may not accept --mcp-config with a "type":"http" server + headers. Mitigation: verify against the installed CLI first (T5 gate); fall back to SSE transport, then to a thin stdio MCP shim that proxies to /api/browser. The stdio shim converges on the REST surface (T3), which is built regardless — so T3 de-risks the whole MCP bet.
  • Dependency weight. Playwright bundles ~300MB Chromium; @modelcontextprotocol/sdk is new (§4.4). Mitigation: justify both in the spec; Chromium fetch mirrors the existing node-pty postinstall step; pin versions.
  • Chromium RAM. Two live sessions ≈ 500MB of browsers alone. Mitigation: MAX_BROWSERS cap with a typed BROWSER_LIMIT_REACHED error (mirrors TERMINAL_LIMIT_REACHED) + idle eviction.
  • MCP endpoint drives a browser. The server binds all interfaces (server.listen(config.port), index.ts:86, no host), so /mcp/browser would be LAN-reachable. Mitigation: require the per-session token on every MCP call and reject non-loopback origins; recommend a follow-up to bind 127.0.0.1 globally.
  • SPA fallback swallows /mcp. Production fallback /^(?!\/api).*/ (index.ts:42) returns index.html for /mcp/browser. Mitigation: register MCP before the fallback (it already runs in registerRoutes before mountBuiltFrontend) and widen the exclusion to /^(?!\/api|\/mcp).*/.
  • Windows process-tree teardown. A plain .kill() may orphan chrome.exe children. Mitigation: prefer Playwright context.close(); fall back to taskkill /pid X /T /F on win32; reap on all shutdown signals.
  • Input contention. Human click mid-agent-action breaks a test. Mitigation: soft lock — human interaction sets controller=human for a grace window; agent tool calls during it return a typed BROWSER_HUMAN_DRIVING the agent can retry; UI shows who's driving with a "take control" affordance.
  • Screencast bandwidth/CPU. Mitigation: screencast only while a viewer is attached (start on attach, stop on detach); JPEG quality + max frame rate as tuned constants.
  • Orphaned profile dirs after a crash. Mitigation: startup GC of profile dirs with no matching session; wipe-on-delete.

Blast radius (existing code touched): ClaudeProcessService (spawn args + env — flows through every session; additive, guarded), SessionsController.remove (one added call), src/index.ts (WS attach + fallback regex), src/routes/index.ts (one mount), src/config/constants.ts (additive), frontend SessionPanel.tsx (layout split), package.json ×2 (deps), repo .gitignore (state dir). Everything else is new files.

Pushback

  • You chose MCP over my recommended skill+REST path. MCP is genuinely the better agent UX (native tools vs shelling out), so this is a defensible call — but it is net-new architecture with a CLI-version dependency, which is why this is Level 3 not Level 2. I have de-risked it by making the REST surface (T3) load-bearing on its own (the UI needs it) and the MCP fallback substrate — so if HTTP MCP proves unsupported, we lose native-tool ergonomics, not the feature.
  • Recommended alternative if T5 verification fails: ship v1 with the browser fully functional for the human plus the REST API, and expose agent control through a stdio MCP shim (or the injected skill) rather than blocking the release on HTTP MCP.
