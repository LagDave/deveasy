Why

To work in a project without leaving DevEasy, the operator needs real terminals — run builds, git, scripts, anything. They must support multiple instances and, critically, survive a browser reload: a long-running command should keep going and the view should reattach. They only die when the DevEasy server stops.

What

The TERMINAL tab (placeholder from plan 1) hosts multiple terminal instances per project, each a real PTY (pseudo-terminal) running the operator's shell in the project directory, rendered with xterm.js. The PTYs live in the backend process — keyed by id, kept in memory for the server's lifetime, killed only on server shutdown. The browser connects to a PTY over a WebSocket; on reload it re-lists the project's live terminals, reconnects, and replays scrollback so the recent output reappears.

Done when: you can open several terminals in a project, run commands, reload the page and find the same terminals still running with their recent output restored, and the PTYs only terminate when the DevEasy server is stopped.

Context

Relevant Files

  • src/services/ClaudeProcessService.ts — the persistence pattern to mirror: an in-memory Map keyed by id, idempotent start, clean teardown, bindSignals() reaping children on SIGINT/SIGTERM, API-key-scrubbed env. The terminal service is the same shape for PTYs.
  • src/ws/sessionWebSocket.ts — the noServer WS upgrade pattern; add a parallel handler for /ws/terminal that does not collide with /ws/sessions.
  • src/index.ts (the <deveasy:websockets> marker) — where the new WS relay attaches.
  • frontend/src/hooks/useSession.ts — the frontend WS-lifecycle hook to model the terminal socket hook on (connect, reconnect, cleanup).
  • src/controllers/git/ + frontend/src/api/git.ts — the controller/route + frontend-triad shapes to copy for the terminal REST management endpoints.
  • frontend/src/components/CodeEditor/TerminalTab.tsx — the placeholder from plan 1, rebuilt here.

New Dependencies (§4.4 justification)

  • node-pty (backend) — the standard native PTY binding. No pure-JS alternative gives a real TTY; the pipe-based fallback was explicitly rejected (vim/top/colors break). Native module: needs prebuilt binaries or a build toolchain (node-gyp) on each machine — see Risk.
  • @xterm/xterm + @xterm/addon-fit (frontend) — the standard terminal emulator widget + auto-resize addon.

Patterns To Follow

  • §5.2 — spawn the shell via node-pty.spawn(shell, [], { cwd, env }) with an argv array and the project path as cwd — never interpolate the path into a shell command string.
  • §7.1-7.4, §8.1-8.3 — the REST management endpoints (create/list/kill) go through Routes → Controller → Service with the { success, data, error } envelope and a typed TerminalError.
  • §9.1 — Pino only; PTY output is never logged (it can contain secrets the operator types) (§5.3).
  • §13.1 — split frontend into TerminalTab (instance strip) + TerminalInstance (one xterm) + useTerminalSocket hook.

Reference file (backend): ClaudeProcessService.ts for the process registry + signal reaping; sessionWebSocket.ts for the WS upgrade. Reference (frontend): useSession.ts for the socket lifecycle.

Constraints

Must

  • PTYs persist in a backend registry for the server's lifetime; closing the WebSocket detaches, it does not kill the PTY.
  • Bound scrollback per terminal (ring buffer, fixed byte cap) replayed on attach.
  • Reap all PTYs on SIGINT/SIGTERM (reuse the bindSignals pattern).
  • Cap concurrent PTYs (mirror MAX_CONCURRENT_SESSIONS).
  • Support resize (cols/rows) from the client via the fit addon.
  • On reload, list the project's live terminals and reconnect to each.

Must Not

  • No unbounded scrollback buffer (memory growth) — cap it (§ failure-mode thinking).
  • Do not log PTY input/output (§5.3).
  • Do not kill PTYs on WS close or browser reload — only on server shutdown.
  • Do not reuse the /ws/sessions channel — separate /ws/terminal.
  • No auth/remote exposure changes — this stays a localhost single-operator app.

Out Of Scope

  • Persisting PTYs across a server restart (they die with the server — that is the stated boundary).
  • Splits/panes inside one terminal, themes, font config (defaults only in v1).
  • Remote/multi-user access or terminal sharing.

Risk

Level: 4 — Major impact. New native dependency, a new WS channel, in-memory process state, and by design arbitrary shell execution at the server's privilege.

Risks Identified

  • Arbitrary command execution. A PTY is a full shell at the server user's privilege. Mitigation: acceptable for a local-first single-operator app bound to localhost; document it. Do not expose the server beyond localhost. No new attack surface vs. the operator's own terminal.
  • node-pty native build. Needs prebuilt binaries or node-gyp + toolchain per machine; can fail on npm install. Mitigation: pin a version with darwin/arm64 prebuilds; document the toolchain fallback; fail fast with a clear message if the binding can't load (§5.6 spirit).
  • Orphaned PTYs / leak. Mitigation: bindSignals reaping + a concurrent-PTY cap + explicit kill endpoint.
  • Scrollback memory growth. Mitigation: fixed-size ring buffer per terminal; drop oldest bytes.
  • Reconnect races / duplicate streams on reload. Mitigation: model the socket hook on useSession; one WS per instance; idempotent attach; replay-then-live ordering.
  • WS upgrade collision. Mitigation: path-scoped noServer handler that ignores non-/ws/terminal upgrades, exactly as sessionWebSocket does for its path.

Blast radius: new TerminalProcessService, new terminalWebSocket.ts, index.ts (attach), new terminal controller/routes, package.json + frontend/package.json (deps), new frontend api/terminal.ts + hooks + TerminalTab children.

Pushback

  • This is genuinely a Level 4 change and the largest of the three. Recommend executing it last, after plans 1 and 2 are merged and stable, and on its own branch — the native dependency alone can derail an install. Confirmed sequencing: plan 1 → plan 2 → plan 3.
  • Consider whether "survives reload" is enough, or whether you'll later want "survives server restart" (would require detaching PTYs from the node process — e.g. tmux/screen-backed — a much bigger design). v1 stops at reload, by decision.

Management API + WS protocol

ChannelShapePurpose
POST /api/terminal{ projectId } → { terminalId }Spawn a PTY in the project cwd
GET /api/terminal?projectId=→ { terminals: [{ id, createdAt }] }List live PTYs (for reattach on reload)
DELETE /api/terminal/:id→ { ok }Kill one PTY
WS /ws/terminal?terminalId=attach → replay scrollback → live duplexStream one PTY

WS messages: client→server { type: "input", data } and { type: "resize", cols, rows }; server→client raw output frames (data) plus a leading scrollback replay.

Revision Log

Rev 1 - 2026-07-01

Change: Executed. Status → In Progress (service + WS relay built and proven by automated tests; UI reload/resize/shutdown acceptance pending a running app). Added scripts/fix-node-pty.mjs + a postinstall hook.

Reason: Execution deviation: node-pty 1.1.0's prebuilt darwin spawn-helper shipped without the execute bit, so posix_spawnp failed at spawn time. The postinstall restores it (no-op off darwin) so npm install yields a working terminal — satisfying the "installs cleanly" Done item.

Note: The PTY child env scrubs ANTHROPIC_API_KEY so claude in the terminal uses the subscription login, consistent with the app's sessions. The Vite dev proxy already forwards all /ws (covers /ws/terminal), so no proxy change was needed. Added an end-to-end WS integration test (terminalWebSocket.test.ts) beyond the spec's task list.

Rev 2 - 2026-07-01

Change: Terminal font (User QA). Added a terminal-only --font-terminal token + @font-face for Comic Code Ligatures, bundled from frontend/public/fonts/ (runtime-served so a missing file falls back to the mono stack, never breaks the build). The xterm instance now uses --font-terminal with letterSpacing: 0 and refits on document.fonts.ready.

Reason: User QA: the JetBrains Mono terminal looked off with loose spacing; switch to bundled Comic Code with tight tracking.

Note: Comic Code is a commercial font, so its files are not committed — drop ComicCodeLigatures-Regular.woff2 / -Bold.woff2 into frontend/public/fonts/ (see that folder's README). Until then the terminal falls back to JetBrains Mono.

Rev 3 - 2026-07-01

Change: Switched the terminal font from Comic Code to MesloLGL Nerd Font (bundled, free). Replaced the Comic Code @font-face with Meslo (regular/bold/italic/bold-italic .ttf committed under frontend/public/fonts/) and repointed --font-terminal.

Reason: User QA: the terminal still read with wide letter spacing (the Comic Code .otf files had landed in src/fonts/, so the @font-face never loaded and it fell back to the wide JetBrains Mono). Meslo LG is a narrower face and a Nerd Font, so it also fixes the missing powerline/git glyphs (the □ boxes in the prompt).
