Why

Sessions can already drive a browser; they can't drive terminals. So "run this app locally and open it" only works if the servers happen to be running already — the agent has no managed, visible way to start them. This makes the session the manager of both its browsers and its terminals, and adds a skill that reliably starts a full-stack app and opens it.

Confirmed live: a prior session transcript shows the agent successfully calling the browser MCP tool to open an app — the agent↔/mcp link works. This plan reuses that proven surface for terminals.

What

Terminals gain an optional sessionId. Created from a project folder (today's path) they stay project-only. Created for a session (by the agent's new terminal tools, or by the operator from the session pane) they are session terminals: they appear in both the project TERMINAL tab and a tabbed terminal strip below the browser in the session pane, and are closed when the session is deleted. A unified /mcp server exposes browser_* + terminal_* tools, and a run-and-preview skill orchestrates: detect scripts → start backend + frontend in session terminals → wait until the port answers → open the browser to the app. Done when: asking a session to "run this app and open it" starts both servers in visible terminals (tabbed below the browser), waits for readiness, and lands the browser on the running app — and deleting the session tears its terminals down.

Decisions locked (grill, 2026-07-02): Run mechanism = visible terminals via a terminal MCP tool (not the agent's background shell). Terminal model = optional sessionId, default null = project-only. Pane = browser (top) + terminal strip (bottom), each region shows when active; the pane opens when either exists. Cleanup = delete only (stopping a session leaves its terminals running). Both the operator and the agent can CRUD session terminals.

Constraints

Must

  • sessionId is nullable; terminals created without it behave exactly as today (project-only).
  • Session terminals appear in BOTH the project TERMINAL tab and the session strip.
  • Session delete closes the session's terminals (session stop does not).
  • Terminal MCP tools reuse the existing loopback + per-session token guard (§5.4); create session terminals in the session's project.
  • Keep TerminalProcessService under the size ceiling (§2.4) — extract if the additions push it past ~800.
  • Tests for the service (session list/close), the terminal MCP tools, and the delete hook (§20.1, §20.2).

Must Not

  • Do not change existing project-terminal behavior or the TERMINAL tab.
  • Do not run servers in the agent's hidden background shell — session terminals only (visibility + lifecycle).
  • No new DB tables — terminals stay in-memory like today (only their session association is added).
  • Do not weaken the MCP auth or the API-key scrub.
  • No second MCP endpoint — one /mcp server for the session's whole workspace.

Out of scope (v2)

  • Persisting terminals across a DevEasy restart (still in-memory).
  • Resizable browser/terminal split ratio (fixed split v1).
  • Generic "run any stack" detection beyond the project's package.json scripts.

Risk

Level: 3 — extends the shared TerminalProcessService (used by the live TERMINAL tab), renames the MCP surface, and adds a new pane layout + an injected skill.

Risks identified

  • Blast radius on the terminal stack. The TERMINAL tab, /api/terminal, and /ws/terminal all use TerminalProcessService. Mitigation: sessionId is additive + nullable; list(projectId) is unchanged; existing tests must stay green.
  • MCP rename touches the working browser feature. Mitigation: a called-out, contained refactor — path constant + config line + server name; browser_* tool names keep "browser" so the browser auto-reveal still matches. Verify a browser tool call still works after the rename (regression gate).
  • Agent starts a long-running server via a terminal tool. terminal_run writes a command and returns immediately (PTY keeps running) — it must not block on the server staying up. Readiness is a separate poll (terminal_read or an HTTP check). Mitigation: the skill polls; the tool never awaits process exit.
  • Auto-reveal false-positives. Detection must distinguish browser_* vs terminal_* so a terminal call doesn't spuriously trigger browser-only logic. Mitigation: key on the tool base name.
  • Concurrency. Two servers per session × sessions can exhaust the MAX_TERMINALS cap. Mitigation: keep the cap; the skill reuses existing terminals when present rather than spawning duplicates.

Blast radius: TerminalProcessService (+ its consumers: TERMINAL tab, /api/terminal, /ws/terminal), SessionsController.remove, the MCP server + its path constant + ClaudeProcessService config, SessionPanel layout. New: terminal MCP tools, session terminal UI, the skill.

Pushback

  • The MCP rename is the one debatable bit. Keeping /mcp/browser and bolting terminal tools onto a "browser" server would avoid the churn but leaves a misnamed surface and messy tool namespaces (mcp__deveasy-browser__terminal_run). Renaming to deveasy//mcp is the correct abstraction for "session workspace control" and the churn is small — recommended. If you'd rather not touch the browser wiring at all, we keep the old name and accept the smell.
