Why

Today a project only appears in DevEasy if you manually git clone a repo into projects/. There is no way to start a fresh, on-standard project from the app. This adds a guided, agent-driven "Create new project" flow on the Projects page that scaffolds a full-stack TypeScript project matching the team's Vision stack — so every new project starts consistent (structure, lint, tsconfig, ports, optional Microsoft auth) instead of hand-assembled.

What

A "Create new project" entry on the Projects page opens a wizard modal. The user types a project name; the backend creates projects/<slug>/, runs git init, injects the shared config, and opens a normal Claude CLI session against it seeded to run a new create-new-ts-project skill. The skill drives a question-and-answer conversation: it emits each question as a fenced deveasy-question JSON block, the wizard renders the choices as clickable buttons, and the selection is sent back into the same session. When the user has answered (name confirmation, Microsoft auth yes/no, etc.), the agent scaffolds the boilerplate file tree into the project directory and signals completion. The new project then appears in the Projects list.

Done when: from the Projects page a user can name a project, answer the wizard's questions via rendered buttons, and end up with a populated, git-initialized projects/<slug>/ (backend + frontend boilerplate, lint/tsconfig, optional MSAL auth scaffolding) that shows up as a selectable project — with no API key required (subscription auth preserved) and no dependency install in this flow.

Context

How the engine works today (grounding)

  • 100% CLI-based. src/services/ClaudeProcessService.ts spawns claude -p --output-format stream-json --input-format stream-json --include-partial-messages --verbose, cwd = project path, and scrubs ANTHROPIC_API_KEY to force subscription auth. We reuse this verbatim — no Agent SDK, no API key.
  • Sessions own a long-lived process. src/controllers/sessions/feature-services/SessionStreamService.ts bridges the process to the WebSocket at /ws/sessions?sessionId=N. Client sends { type: "user_turn", text }; events stream back (stream_event, assistant, result, …). The wizard reuses this channel unchanged.
  • Config injection. src/services/ConfigInjectionService.ts symlinks CLAUDE.md + .claude/skills + .claude/agents into a project so the CLI's config walk finds them. A new skill in .claude/skills/<name>/SKILL.md is automatically visible to every project.
  • Projects are scanned, not created. src/controllers/projects/feature-services/ProjectScanService.ts lists direct children of projects/ that are git repos and upserts them into the DB. GET /api/projects is the only endpoint. src/services/PathGuard.ts rejects names with / \ .. and anything outside the root.
  • Chat renders markdown only. frontend/src/components/Session/ + frontend/src/hooks/useSession.ts flatten events to markdown bubbles; SessionComposer.tsx forces a command prefix on typed input. There is no interactive/choice rendering yet.

Wizard JSON Protocol (the contract)

The skill instructs the agent to ask questions only by emitting exactly one fenced block per turn. The frontend parses the block client-side; the backend stays untouched (selections ride the existing user_turn channel as readable text). Parsing is tolerant — if no block is found the assistant text renders as plain markdown, and a free-text reply box is always available as a fallback.

A question block

```deveasy-question
{
  "id": "auth",
  "prompt": "Do you want Microsoft (Entra) sign-in scaffolded?",
  "type": "single",            // "single" | "multi" | "text"
  "options": [
    { "value": "msal", "label": "Yes — scaffold MSAL auth", "hint": "@azure/msal-node + msal-browser, env placeholders" },
    { "value": "none", "label": "No auth for now" }
  ],
  "allowOther": false           // if true, the wizard also shows a free-text box
}
```

The completion block

```deveasy-done
{ "project": "acme-portal", "summary": "Backend + frontend scaffolded. Microsoft auth: yes. Run npm install in each package to start." }
```

On a button click the wizard sends a user_turn whose text is the human-readable label(s) plus the machine value, e.g. Yes — scaffold MSAL auth [auth=msal], so the agent reads the choice naturally. On deveasy-done the wizard invalidates the projects query and offers to close.

Constraints

Must

  • Reuse the existing CLI + WebSocket + injection pipeline; preserve subscription auth (never set ANTHROPIC_API_KEY).
  • Validate the project name through PathGuard (§5.2) before any filesystem op; slugify, reject collisions with an existing dir/DB row.
  • Backend follows §6.1/§7.x/§8.x: thin controller, logic in ProjectCreateService, DB only via models, standard envelope, Pino logs.
  • Skill embeds the full Vision-stack boilerplate inline and emits questions via the deveasy-question protocol.
  • Skill explicitly suspends the injected CLAUDE.md command gate for this flow and forbids the AskUserQuestion tool — questions go through JSON blocks only.
  • Frontend follows the triad (§12.1, §14.x, §15.1, §16.1); no any/console/raw fetch.

Must Not

  • Add the Agent SDK or any new runtime auth path.
  • Reference the projects/Vision folder from the skill — the structure must be self-contained in the skill text.
  • Run npm install or start dev servers in this flow (files + git init only).
  • Write secrets/real tenant IDs into the scaffold — env placeholders + .env.example only (§5.1, §17.3).
  • Change the WebSocket/session protocol or the command-gate behavior of normal sessions.

Out Of Scope

  • Dependency install, dev-server launch, and port allocation at runtime (documented as next steps in the generated README).
  • Deleting/renaming projects, or a cancel-cleanup that removes a half-created project dir (the empty git repo is valid and user-deletable; can be a follow-up).
  • Templates for stacks other than the Vision/team stack.
  • Editing the skill from the Agent Manager UI (it works automatically since it lives in .claude/skills/, but no special UI is added).

Risk

Level: 3 (Structural — introduces a filesystem-mutating endpoint and an agent that writes files + git-inits into the workspace).

Risks Identified

  • R1 — Path traversal / arbitrary dir creation (Level 3, security §5.2). The create endpoint mkdirs from a user-supplied name. Mitigation: route the name through PathGuard.resolveProjectPath exactly as the scan path does; slugify to [a-z0-9-]; reject if the resolved path already exists or escapes projectsRoot.
  • R2 — Injected command gate fights the wizard (Level 3, correctness). The symlinked CLAUDE.md forces a command prefix on every turn and offers AskUserQuestion; either would derail the JSON-block conversation. Mitigation: the seed turn opens with a valid command to pass the gate, and the skill explicitly overrides — suspend the gate for this flow, never call AskUserQuestion, ask only via deveasy-question blocks, and wait for the user's answer turn between questions.
  • R3 — Choice JSON unreliable / malformed (Level 2). The model may emit invalid or missing JSON. Mitigation: strict skill instructions (one block, fenced, exact schema) + a tolerant parser that falls back to rendering plain markdown, with a free-text reply box always present.
  • R4 — Orphan project on abandonment (Level 2). The DB project + session rows and the dir are created before scaffolding finishes; abandoning leaves an empty project. Mitigation: the dir is a valid empty git repo and re-scan-safe; accept for v1, note cleanup as out-of-scope follow-up.
  • R5 — Concurrency slot (Level 1). The wizard consumes one of the 8 concurrent CLI slots like any session. Mitigation: none needed; identical to normal sessions.

Blast radius: new POST /api/projects on the existing projects router (additive; GET unchanged). New ProjectCreateService. New frontend feature folder + two additive functions in api/projects.ts/useProjects.ts. ProjectPicker.tsx gains an affordance. Reuses useSession, ConfigInjectionService, ProjectModel, SessionModel, PathGuard with no behavior change. New skill file is inert until invoked.

Pushback

  • "Agent SDK" was the stated ask, but it fights the architecture. DevEasy is deliberately subscription-only (API key scrubbed). The Agent SDK would either require a paid API key or just re-wrap the CLI. Recommendation (accepted): drive the wizard through the existing CLI pipeline; the skill provides the "agentic" behavior. Revisit the SDK only if subscription auth is ever dropped.
  • Boilerplate baked into a skill will drift from Vision over time. A markdown template is not tested like code. Recommendation: treat the skill as the single source of truth for "a new team project," version it in git (auto-committed via the Agent Manager path), and revisit if drift bites. Acceptable for v1.
