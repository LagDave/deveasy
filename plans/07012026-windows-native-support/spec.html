Why

DevEasy only runs cleanly on macOS/Linux today. A Windows teammate cannot start it: npm run dev:all dies immediately, and even past that, the core feature that makes DevEasy itself work — injecting the shared agent config into each project — relies on POSIX symlinks that a default non-admin Windows user cannot create. The goal is end-to-end native Windows support (PowerShell/cmd, no WSL2 required) without changing any macOS/Linux behavior.

What

On a clean native Windows box (Node 20+, Docker Desktop, Claude CLI on PATH, git), a user can clone the repo, npm install, copy .env, and run npm run dev:all to reach the app on http://localhost:1234 — start a Claude session, open a terminal, and have the shared CLAUDE.md + .claude/skills + .claude/agents injected into projects without admin rights or Developer Mode. Config edits in the Agent Manager still propagate live into open projects. macOS/Linux behavior is unchanged.

Context

Relevant Files

  • scripts/dev.mjs — one-command dev launcher. Two Windows breaks: spawnSync("sh", ["-c", "sleep 2"]) (no sh on Windows) and run() spawning npm with shell:false (npm is npm.cmd — ENOENT without a shell on Windows).
  • src/services/ConfigInjectionService.ts — injects shared config via fs.symlink (line 119). Symlinks need admin/Developer Mode on Windows. Idempotency check (line 104) uses isSymbolicLink(), which will not recognize a hardlink.
  • src/services/ClaudeProcessService.ts — spawns bare claude (lines 82, 125, 197). On Windows the binary is claude.cmd; spawn/spawnSync with shell:false cannot find it. runHeadless passes an arbitrary prompt as an arg — must not be shell-interpreted (§5.2).
  • src/services/TerminalProcessService.ts — PTY shell fallback to powershell.exe (line 19); cwd not validated before spawn; SIGINT/SIGTERM reaping only (line 251).
  • src/index.ts — process shutdown on SIGINT/SIGTERM (lines 96–97). Windows console signals differ.
  • src/services/AgentConfigService.ts — writes the shared CLAUDE.md in place (line 266, fs.writeFile to the same path, no atomic rename). Verified: a hardlink to it survives saves, so live-update is preserved with the hardlink approach.
  • .env.example, README.md — setup docs; no Windows section today.

Patterns To Follow

  • Typed domain errors with a machine code (§8.3) — mirror TerminalError / ProjectError already used in these services.
  • Pino childLogger for all logging in src/ (§9.1). scripts/dev.mjs is a build script, not production src/, so its process.stdout.write usage stays.
  • New shared helper lives in src/utils/ (§6.2) — currently empty; this is the first util. Single responsibility (§2.1), small (§2.4), no any (§4.5), named constants for PATHEXT defaults (§4.2).
  • Branch on process.platform === "win32"; the POSIX path must stay byte-for-byte identical to today.

Reference file: src/services/ConfigInjectionService.ts — modified in place; its existing structure (typed errors, idempotent guards, JSDoc rationale) is the analog for the new branches.

Constitution note: this repo does not have check:conventions/check:all wired (the Constitution references them but they are absent here). Verification uses the real scripts: npm run typecheck, npm run lint, npm run test. Articles are cited manually.

Constraints

Must

  • Keep macOS/Linux behavior byte-for-byte (POSIX branch unchanged).
  • Inject config on Windows with no admin / no Developer Mode: junctions for the two directories, hardlink for CLAUDE.md.
  • Preserve live config propagation (hardlink + in-place write — §verified).
  • Keep shell:false on the Claude spawn paths; resolve the executable instead (§5.2 — no arbitrary prompt through a shell).
  • Typed errors + Pino logging on every new async path (§3.1, §3.2, §3.3, §8.3).
  • New util has a mirroring test file (§20.1).

Must Not

  • No new runtime dependencies — Node stdlib only (§4.4).
  • No shell:true on the claude spawn sites.
  • Do not refactor unrelated code; no drive-by cleanups.
  • Do not change the response/error contract or the config-tree path guard (§5.2).

Out Of Scope

  • WSL2 instructions (chosen path is native; WSL2 not documented here).
  • Native (non-Docker) PostgreSQL install path — Docker Desktop is assumed.
  • Windows installer / packaging / service registration.
  • Wiring up check:conventions — separate concern.
  • Frontend changes — none needed for Windows.

Risk

Level: 3 — Structural. The config-injection mechanism is the heart of DevEasy's sharing model and is changed across platforms; the change is also cross-cutting (launcher, two process services, shutdown). Mitigation path (junction + hardlink, no privilege) was chosen in planning.

Risks Identified

  • Hardlink needs same volume. If PROJECTS_ROOT is on a different drive than the repo, fs.link throws EXDEV. Mitigation: catch EXDEV, fall back to a file copy, and log a warning that edits won't live-update until re-injection. Document the same-drive recommendation.
  • Hardlink idempotency / false conflict. A hardlink looks like a plain file to lstat, so the current check would throw CONFIG_INJECTION_CONFLICT on re-inject. Mitigation: on Windows, compare fs.stat(linkPath).ino against the shared file's inode — equal means it's our link (skip), different means the project owns a real CLAUDE.md (genuine conflict).
  • Junction detection. Node reports junctions through lstat().isSymbolicLink() and readlink(), but this needs confirming on a real Windows box. Mitigation: acceptance item verifies idempotent re-inject on Windows.
  • Editor atomic-save breaks a hardlink. If the shared CLAUDE.md is ever replaced by rename (not in-place), the hardlink detaches. Mitigation: DevEasy's own writer is in-place (verified, AgentConfigService:266); document the caveat for external editors.
  • Orphaned children on console close. Windows cannot reliably trap a console-window close (CTRL_CLOSE). Mitigation: handle SIGINT (Ctrl+C) and SIGBREAK (Ctrl+Break) to reap children; document the close-button limitation. Full job-object containment is out of scope.

Blast radius: ConfigInjectionService.inject is called from ProjectCreateService.ts (create-project wizard) and SessionsController.ts (session start) — both high-traffic, both must keep working on macOS. ClaudeProcessService consumers: SessionStreamService.ts, SessionTitleService.ts (runHeadless), sessions/feature-utils/controllerResponses.ts. TerminalProcessService consumers: TerminalController.ts, ws/terminalWebSocket.ts (+ existing tests). src/index.ts shutdown affects the whole process.

Pushback

  • The copy fallback silently degrades the live-sharing promise on a cross-drive PROJECTS_ROOT. Recommended: keep the fallback but make the warning loud, and state the same-drive recommendation in the README rather than trying to engineer cross-volume live links.
  • I cannot verify any of this from macOS. Acceptance items are concrete Windows steps for you to run; the plan is honest that Layer-2 stays pending until then.
