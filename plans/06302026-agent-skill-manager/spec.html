Why

The shared agents, skills, and CLAUDE.md are the team's common workflow. They must be editable in-app and committed to version control so teammates stay in sync by pulling. This slice gives a simple editor over those files and commits changes — keeping every operator on the same agents without hand-editing files.

What

A CRUD editor for the DevEasy-root config that gets injected into projects: subagent files in .claude/agents/, skill files in .claude/skills/, and the root CLAUDE.md. Each is markdown with YAML frontmatter; the editor exposes the frontmatter fields and the markdown body, and saving writes the file and commits it to the DevEasy repo with the configured author. Done when: a user can list, create, edit, and delete a subagent or skill in the UI, save it, and see a commit in the DevEasy repo containing exactly that change — which a teammate then gets by pulling.

Patterns To Follow

  • Backend domain shape mirrors gbp-automation/ (§6.1): thin controller, logic in services, envelope + typed error. This domain has no DB — it is a filesystem + git domain, so there is no model; that is acceptable because there is no DB access to route (§7.4 applies only to DB calls).
  • Frontend triad (§12.1); list/content is React Query server state with invalidation on save (§15.1).

Reference analogs: Constitution §6.1, §12.1. Reuses the active-project-agnostic config at the DevEasy root from Spec 1; independent of Specs 2 and 3.

Constraints

Must

  • §5.2 hard path guard: every read/write resolves inside .claude/agents, .claude/skills, or the root CLAUDE.md — reject anything else
  • Preserve frontmatter validity: parse and re-serialize with gray-matter; validate required fields before write
  • Commit author = Project Profile (git -c user.name=... -c user.email=...); commit message states the file changed
  • §8.1 envelope; §9 Pino; §3 handle every async FS/git call
  • §14.2 all frontend HTTP through api/index.ts

Must Not

  • Do not store agent/skill content in the DB — files are the source of truth (Spec 1 boundary)
  • Do not auto-push — commit locally only; the user pushes manually (matches v1 git decision)
  • Do not write outside the config tree; no arbitrary file editor
  • No in-app agent test-runner in v1 (keep it simple)

Out Of Scope

  • Auto-push / PR for config changes, conflict resolution UI, diff viewer
  • Codex AGENTS.md management (Claude only for v1)
  • Testing an agent before committing

Risk

Level: 3 — this edits config that is injected into every project and is committed to version control, so a bad write affects the whole team after a pull.

Risks Identified

  • Arbitrary file write / path traversal. A crafted filename could escape the config tree. Mitigation: resolve the absolute path and assert it is inside the whitelisted config directories before any write/delete (§5.2); reject path separators in names.
  • Corrupt frontmatter breaks every session. Since these files load into Claude, malformed YAML degrades all projects. Mitigation: parse + validate required frontmatter fields before writing; reject invalid documents with a clear error rather than committing them.
  • Noisy or wrong-author commits. Mitigation: one commit per save scoped to the changed file, message naming it, author forced to the profile identity; never amend or force.
  • Editing live, injected symlinks. Because Spec 1 symlinks these into open projects, an edit is immediately visible to active sessions. Mitigation: acceptable and intended (that is the sync mechanism); note it in the UI so edits aren't surprising mid-session.

Blast radius: writes to the DevEasy repo's config tree + local commits. Consumers: every project via Spec 1 injection, every teammate after a pull. No new DB tables.

Pushback

  • No diff/preview before commit is a sharp edge for a shared artifact. Recommended minimum: show the rendered file and require an explicit Save. A proper diff viewer is a fast-follow, not v1.
