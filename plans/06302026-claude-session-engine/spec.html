Why

The whole point of DevEasy is chatting with the Claude Code agent against the selected project — using the user's own subscription, not API credits. Verified constraint: the Agent SDK cannot use subscription auth, but the CLI can. So the engine spawns the real CLI in structured streaming mode and builds a custom chat UI on the event stream. This is the product's core loop.

What

A backend that, for the active project, spawns claude -p --output-format stream-json --input-format stream-json --verbose with cwd set to the project and ANTHROPIC_API_KEY scrubbed from the environment (so the cached subscription login is used). It relays the NDJSON event stream to the browser over WebSocket, accepts user turns back, and persists the full transcript to Postgres. The frontend renders a streaming chat UI: user/assistant turns, tool-use and tool-result events, and final results. Done when: a user can send a message to a selected project, watch Claude respond with streamed tool calls, and reload the page to see the persisted transcript.

Patterns To Follow

  • Backend domain mirrors gbp-automation/ (§6.1): thin controller, logic in feature-services/, DB only via models, typed error + envelope.
  • Worker-style discipline for the long-lived process: idempotent start, clean teardown, failures logged with context (§21 spirit, applied to the process manager).
  • Frontend chat reads from a hook, not inline; errors surfaced via toast/error state (§14.3, §16).

Reference analogs: Constitution §6.1 (backend skeleton), §12.1 (frontend triad). Depends on the active-project state and shell from Spec 1.

Constraints

Must

  • Subscription auth: spawn with ANTHROPIC_API_KEY deleted from the child env; never inject an API key
  • §7.4 all transcript persistence through models; §8.1 REST endpoints use the envelope
  • §3.1/§3.2 every async/stream handler wrapped; never swallow a process error
  • §9.1/§9.3 Pino with session + project context on every log line
  • Stream parsing tolerant of partial NDJSON lines (buffer until newline)
  • §14.2/§15.1 WS data flows through a hook; persisted transcript fetched via React Query

Must Not

  • Do not store config (agents/skills) in the DB — only session/message rows (§ Spec 1 boundary)
  • Do not run more than one live process per session; cap concurrent sessions
  • Do not log message content that could contain secrets verbatim at info (§5.3) — store in DB, log identifiers
  • No raw fetch in components (§14.2)

Out Of Scope

  • Git/Azure cockpit (Spec 3), agent editor (Spec 4)
  • Multi-user / shared sessions — local-first, single operator per instance
  • Resuming a CLI session across app restarts (persist transcript only; live process is ephemeral)

Risk

Level: 3 — long-lived child processes, auth-mode correctness, and stream parsing.

Risks Identified

  • Wrong auth mode silently billing API credits. If ANTHROPIC_API_KEY leaks into the child env, the CLI bills pay-as-you-go instead of the subscription. Mitigation: explicitly delete env.ANTHROPIC_API_KEY in the spawn, and assert at startup that the CLI is logged in (surface a clear "log in with claude first" state if not).
  • Orphaned processes. A crashed tab or server could leave CLI processes running. Mitigation: track child PIDs in a registry, kill on session close / server shutdown, bound max concurrent sessions.
  • Partial NDJSON / backpressure. stream-json arrives in chunks. Mitigation: line-buffer stdout, parse per complete line, drop-and-log malformed lines rather than crash.
  • CLI version drift changes the event schema. Mitigation: render known event types explicitly and fall back to a raw/JSON view for unknown types so nothing is lost.

Blast radius: new tables + a WS endpoint; consumes Spec 1's active-project. No other slice depends on this.

Pushback

  • An embedded xterm.js terminal would be faster to ship but gives a raw TUI, not the structured UI you want. Recommended: stream-json + custom UI as specced; keep a terminal tab as a later option only if needed.
