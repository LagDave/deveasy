Why

The whole point of DevEasy is chatting with the Claude Code agent against the selected project — using the user's own subscription, not API credits. Verified constraint: the Agent SDK cannot use subscription auth, but the CLI can. So the engine spawns the real CLI in structured streaming mode and builds a custom chat UI on the event stream. This is the product's core loop.

What

A backend that, for the active project, spawns claude -p --output-format stream-json --input-format stream-json --verbose with cwd set to the project and ANTHROPIC_API_KEY scrubbed from the environment (so the cached subscription login is used). It relays the NDJSON event stream to the browser over WebSocket, accepts user turns back, and persists the full transcript to Postgres. The frontend renders a streaming chat UI: user/assistant turns, tool-use and tool-result events, and final results. Done when: a user can send a message to a selected project, watch Claude respond with streamed tool calls, and reload the page to see the persisted transcript.

Patterns To Follow

  • Backend domain mirrors gbp-automation/ (§6.1): thin controller, logic in feature-services/, DB only via models, typed error + envelope.
  • Worker-style discipline for the long-lived process: idempotent start, clean teardown, failures logged with context (§21 spirit, applied to the process manager).
  • Frontend chat reads from a hook, not inline; errors surfaced via toast/error state (§14.3, §16).

Reference analogs: Constitution §6.1 (backend skeleton), §12.1 (frontend triad). Depends on the active-project state and shell from Spec 1.

Constraints

Must

  • Subscription auth: spawn with ANTHROPIC_API_KEY deleted from the child env; never inject an API key
  • §7.4 all transcript persistence through models; §8.1 REST endpoints use the envelope
  • §3.1/§3.2 every async/stream handler wrapped; never swallow a process error
  • §9.1/§9.3 Pino with session + project context on every log line
  • Stream parsing tolerant of partial NDJSON lines (buffer until newline)
  • §14.2/§15.1 WS data flows through a hook; persisted transcript fetched via React Query

Must Not

  • Do not store config (agents/skills) in the DB — only session/message rows (§ Spec 1 boundary)
  • Do not run more than one live process per session; cap concurrent sessions
  • Do not log message content that could contain secrets verbatim at info (§5.3) — store in DB, log identifiers
  • No raw fetch in components (§14.2)

Out Of Scope

  • Git/Azure cockpit (Spec 3), agent editor (Spec 4)
  • Multi-user / shared sessions — local-first, single operator per instance
  • Resuming a CLI session across app restarts (persist transcript only; live process is ephemeral)

Risk

Level: 3 — long-lived child processes, auth-mode correctness, and stream parsing.

Risks Identified

  • Wrong auth mode silently billing API credits. If ANTHROPIC_API_KEY leaks into the child env, the CLI bills pay-as-you-go instead of the subscription. Mitigation: explicitly delete env.ANTHROPIC_API_KEY in the spawn, and assert at startup that the CLI is logged in (surface a clear "log in with claude first" state if not).
  • Orphaned processes. A crashed tab or server could leave CLI processes running. Mitigation: track child PIDs in a registry, kill on session close / server shutdown, bound max concurrent sessions.
  • Partial NDJSON / backpressure. stream-json arrives in chunks. Mitigation: line-buffer stdout, parse per complete line, drop-and-log malformed lines rather than crash.
  • CLI version drift changes the event schema. Mitigation: render known event types explicitly and fall back to a raw/JSON view for unknown types so nothing is lost.

Blast radius: new tables + a WS endpoint; consumes Spec 1's active-project. No other slice depends on this.

Pushback

  • An embedded xterm.js terminal would be faster to ship but gives a raw TUI, not the structured UI you want. Recommended: stream-json + custom UI as specced; keep a terminal tab as a later option only if needed.

Rev 1 - 2026-06-30

Change: Executed. All backend + frontend files built and typecheck-clean; the WebSocket relay attaches on /ws/sessions and REST endpoints respond with the envelope (verified live at boot). The CLI spawn deletes ANTHROPIC_API_KEY from the child env (subscription auth). Reason: Execution deviation: WebSocket uses a noServer server with manual upgrade routing on the pathname (rather than a path-bound server) so additional WS endpoints can attach without conflict. CLI/concurrency constants (MAX_CONCURRENT_SESSIONS) live locally in ClaudeProcessService rather than the shared config/constants.ts, to respect the parallel-build file lane.

Updated Done criteria: Model/process unit tests (§20.1/§20.2) were not written during the parallel build (outside the assigned file lane) and remain open follow-up work.

Rev 2 - 2026-06-30

Change: Chat UI rebuilt for usability. The transcript previously dumped every raw stream-json event (including init and hook_started/hook_progress system noise) as JSON, burying the conversation and pushing the composer off-screen. Now: a classifier (sessionEvents.ts) flattens events into clean render items; assistant/user turns render as message bubbles, tool calls as collapsible cards, and all system/hook events are quarantined behind a single collapsible "N system events" strip. The composer is pinned to the bottom with auto-grow + Enter-to-send, the transcript scrolls and auto-follows, and a "Claude is working" indicator shows while streaming (new streaming flag on useSession, cleared on the result event).

Reason: User QA: "not usable" — the raw-JSON flood made the chat unreadable and the input appeared to do nothing because it was below the fold.

Rev 3 - 2026-06-30

Change: Sessions reorganized into a project-grouped left rail (replacing the dropdown). Each session shows a running indicator — a pulsing dot driven by isLive, derived from the CLI process registry (ClaudeProcessService.getLiveSessionIds()) and polled every 4s via a new useAllSessions query. A session is now started by hitting "+" next to any project, decoupled from the global active-project selection. New backend: GET /api/sessions lists across all projects with isLive annotation (SessionModel.listAll); session create now injects the shared config into the chosen project (best-effort — tolerates CONFIG_INJECTION_CONFLICT so a project's own config wins) so a session no longer requires making the project active first.

Reason: User request: list sessions in a sidebar grouped by project, show a running indicator, and start sessions by selecting a project rather than switching the active project.

Updated Done criteria: Verified live — session create injects config and persists; GET /api/sessions returns all projects' sessions with isLive.

Rev 4 - 2026-06-30

Change: Decoupled the CLI process lifetime from the WebSocket. Previously closing the socket (which happens whenever you switch sessions) killed the other session's process. Now SessionStreamService is a runtime registry: the process lives in the backend independently of any socket, sockets attach/detach as subscribers (multi-subscriber fan-out), and the transcript persists even with zero watchers. A reconnecting socket re-attaches to the running process and is told the current state on the ready event. Runtime state is now working (responding) vs idle (alive, awaiting input), replacing the boolean isLive — the sidebar shows pulsing amber "working", steady green "ready", or grey offline. Added explicit end-of-session via POST /api/sessions/:id/stop + an "End session" button (processes otherwise persist up to the concurrency cap).

Reason: User report: switching sessions cancelled the other session. The fix is a long-running-process model, not a job queue.

Pushback (recorded): A BullMQ/Redis worker was requested but is the wrong primitive — it models discrete run-to-completion jobs, whereas a Claude session is a long-lived interactive process that pauses for human input. The in-memory process registry + DB-persisted transcript already deliver background continuation, reattach, and the working/awaiting-input indicator. Redis/BullMQ would only matter for multiple backend instances (not the local-first model); a child process cannot survive a server restart regardless, so it would not add durability here.

Verified: live lifecycle test — process stays idle after the socket closes (switch), and goes null only on explicit stop.

Rev 5 - 2026-06-30

Change: (1) Runtime state is now driven by the event stream, not just the user's send: any assistant/tool-result event marks the session working, a result marks it idle (system/init events don't change it). This keeps the sidebar indicator correct after a reattach mid-turn, where a session was wrongly showing "ready" while still using tools. (2) The client's streaming flag now tracks the same live activity, so the in-chat "Claude is working" indicator reappears on reattach instead of staying hidden. (3) Opening a session now jumps instantly to the latest message (instant scroll on first paint via a per-session remount key), then smooth-scrolls for new messages.

Reason: User QA: a reopened session showed "ready" while still working, the in-chat working indicator was gone, and opening landed mid-transcript instead of at the bottom.

Rev 6 - 2026-06-30

Change: Assistant messages now render real markdown (react-markdown + remark-gfm) via a themed ui/Markdown renderer — headings, bold/italic, lists, links, tables, inline code chips, and fenced code blocks styled to the dark theme. User turns stay verbatim (whitespace-preserved).

Reason: User QA: assistant text was showing raw markdown source (**bold**, ###, backticks, list markers).

Rev 7 - 2026-06-30

Change: Session management. (1) Auto-naming: on the first prompt of an unnamed session, a fire-and-forget headless CLI call (ClaudeProcessService.runHeadless → SessionTitleService, subscription auth) generates a short title and persists it; the polled sidebar picks it up. (2) Rename: inline edit in the rail → PATCH /api/sessions/:id (1–80 chars). (3) Delete: hover action with confirm → DELETE /api/sessions/:id (stops the process if live; transcript cascades). Deleting the open session clears the selection.

Reason: User request: auto-name sessions from the first prompt, and add rename + delete.

Verified: rename + delete tested live (title updates, empty rejected, row removed); auto-naming wired + builds, pending a live first-turn run.

Rev 8 - 2026-06-30

Change: Token-by-token streaming. Added --include-partial-messages to the CLI args; the backend relays the partial stream_event deltas live but does not persist them (only the final assistant message is persisted, so history replay is unchanged). The client accumulates text deltas into a live "Claude" bubble that fills in as it generates, replaced by the committed message when the full event lands; the working spinner shows only before the first token.

Reason: User request: stream the response as it generates instead of waiting for the whole message.
