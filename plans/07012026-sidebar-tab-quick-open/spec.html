Why

Today a project header in the Sessions rail has only a folder button that opens the editor takeover on its default Files tab, plus a + for a new session. Reaching the Repo or Terminal tab means opening the takeover, then clicking across the tab strip. Adding per-tab shortcut icons on the project header gets the operator straight to the tab they want in one click.

What

Each project header gains hover-revealed compact icon buttons: Files (folder), Repo (branch), Terminal (terminal), and + (new session), in that order. Clicking Repo or Terminal opens the editor takeover with that tab already active; Files keeps the current behavior. The buttons appear only on row hover (like the existing session rename/delete actions), so the resting header stays just the collapse chevron + project name + git line.

Done when: hovering a project reveals the four icons; Repo opens the takeover on the Repo tab, Terminal on the Terminal tab, folder on Files; + still starts a session; collapse toggle and git line are unchanged.

Context

Relevant Files

  • frontend/src/components/Session/SessionSidebar.tsx - renders the project header + action buttons; gains the Repo/Terminal buttons and the hover-reveal layout. Calls onOpenEditor.
  • frontend/src/components/Session/SessionPanel.tsx - owns editorProjectId state and renders CodeEditorView; must also track which tab to open and pass it down.
  • frontend/src/components/CodeEditor/CodeEditorView.tsx - the takeover shell; holds the active-tab useState<WorkspaceTab> (default "files"). Must accept an optional initialTab to seed it.
  • frontend/src/components/CodeEditor/WorkspaceTabs.tsx - exports the WorkspaceTab type ("files" | "repo" | "prs" | "terminal") and the canonical icon-per-tab mapping. Read only — no change.

Patterns To Follow

  • Icon names come from the same mapping WorkspaceTabs uses: folder (Files), branch (Repo), terminal (Terminal), add (new session). Do not invent new icon names.
  • Hover-reveal action buttons mirror the existing SessionRow actions in the same file: a group on the row container and hidden group-hover:flex on the action cluster, with compact grid h-6 w-6 place-items-center rounded text-faint hover:bg-surface-3 hover:text-ink buttons.
  • Tab state stays UI state in the React tree (§15.2) — the requested initial tab is passed as a prop; it is not server state and is not mirrored into React Query or a new store (§15.4).
  • The collapse chevron + name toggle button stays as-is and remains its own click target, separate from the action cluster.

Reference file: frontend/src/components/CodeEditor/WorkspaceTabs.tsx - source of the tab ids and icon-per-tab mapping the new buttons must match.

Constraints

Must

  • Reuse the existing WorkspaceTab type; import it, don't redeclare it.
  • Default behavior unchanged: folder opens Files, + opens a new session.
  • Action buttons reveal on row hover; resting header is chevron + name + git line only.
  • No any (§17.2); no console.* (§17.1); component stays lean (§13.2).

Must Not

  • No new dependencies; no new state library (§15.4).
  • Do not change the editor's tab strip, the tab bodies, or any API/hook.
  • Do not regress the collapse-toggle feature shipped in 0.1.1.
  • No drive-by refactor of unrelated sidebar code.

Out Of Scope

  • A PRs shortcut button (only Files/Repo/Terminal requested).
  • Remembering the last-used tab per project.
  • Deep-linking / URL routing to a tab.

Risk

Level: 2 (Concern — a shared prop signature changes)

Risks Identified

  • Signature change to onOpenEditor. Adding an optional tab arg touches the prop type and the one call site. Mitigation: make the arg optional ((projectId, tab?) => void) so the existing folder call stays valid; TypeScript covers the rest.
  • Initial tab must re-seed each open. CodeEditorView is conditionally rendered, so it remounts on each open and useState(initialTab ?? "files") re-seeds correctly. Mitigation: rely on the existing remount; do not add an effect that fights user tab clicks after open.
  • Four icons in a 288px rail. Mitigation: hover-reveal + compact h-6 w-6 buttons; the name truncates, the cluster is shrink-0.

Blast radius: onOpenEditor has exactly two touch points — its type in SessionSidebar Props and the onOpenEditor={setEditorProjectId} wiring in SessionPanel. CodeEditorView is rendered only by SessionPanel. No backend, API, hook, or type-module change.

Pushback

  • None material. Threading a one-shot initial tab as a prop is the minimal correct change; a router or shared store for a transient takeover would be over-engineering (§15.4).

Revision Log

Rev 2 - 2026-07-01

Change: Dropped hover-reveal — the four action icons are now always visible. Removed the opacity-0/pointer-events-none gate (and the now-dead group class on the header row); the cluster is plain flex shrink-0. The project name keeps flex-1 min-w-0 + truncate, so a long name shows an ellipsis while the icons always stay in place.

Reason: User QA: for long project names the user wants the icons always shown with the title truncated, not gated behind hover. Supersedes the Rev 1 hover behavior (the no-shift goal still holds — always-visible icons can't shift on hover).

Updated Done criteria: The hover-reveal Done item no longer applies; icons are always visible and the title truncates.

Rev 1 - 2026-07-01

Change: The hover-revealed action cluster used hidden group-hover:flex, so it had zero width at rest and the project name reflowed when the icons appeared on hover. Switched to keeping the cluster in layout always and toggling visibility: flex opacity-0 pointer-events-none group-hover:opacity-100 group-hover:pointer-events-auto transition-opacity. Space is now reserved at rest (the name truncates to the same width whether hovered or not), so hover no longer shifts the layout. pointer-events-none keeps the invisible buttons unclickable until revealed.

Reason: User QA: hovering a project header shifted the row layout.

Updated Done criteria: Added the no-layout-shift-on-hover check to the Done list.
