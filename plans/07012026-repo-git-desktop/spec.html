Why

Git is currently read-only inside DevEasy (status, history, branches, checkout). To actually work, the operator needs to inspect changes, commit, switch and create branches, push, and merge to main — without leaving the app. This plan turns the REPO tab (placeholder from plan 1) into a GitHub-Desktop-style working view.

What

REPO becomes a three-region view scoped to the open project: a changed-files list (left), a diff viewer (main, Monaco DiffEditor of HEAD vs working tree), and a commit bar (bottom: message + commit). A toolbar offers branch switch / create, push, and merge-to-main (local merge into main + push). Push and merge reuse the machine's existing git credentials — no new secret storage.

Done when: a changed file shows its diff on click; staging + commit produces a real commit; branch create/switch works; push updates the remote; merge-to-main merges the current branch into main and pushes, with conflicts surfaced (not silently left half-merged).

Patterns To Follow

  • §5.2 (command injection) — every git call is execFile("git", [argv], { cwd }); never a shell string. Branch names validated against the real branch list (as checkout already does) or a strict ^[\\w./-]+$ allow-list; file paths validated within the repo via the existing path guard.
  • §7.1-7.4 — Routes → Controller → GitService. Controllers shape responses only; no git logic in routes/controllers.
  • §8.1-8.3 — { success, data, error } envelope; typed GitError with machine codes (e.g. GIT_MERGE_CONFLICT, GIT_PUSH_FAILED); one error→status mapper; never leak git stderr verbatim to the client (§3.4) — log it, return a generic message.
  • §9.1 — Pino for all logging; no console.*.
  • §16.1 / §16.3 — frontend errors flow through the api client's ApiError and the shared toast.
  • §13.1 — RepoTab will be large; split into ChangedFilesList, DiffViewer, CommitBar, BranchToolbar child components under components/CodeEditor/repo/.

Reference file (backend): GitService.checkout() — copy its branch-validation + argv + getStatus-after pattern for every write op. Reference (controller/error): src/controllers/gbp-automation/ (Part VI). Reference (frontend triad): existing api/git.ts + useGit.ts.

Constraints

Must

  • All git invoked via execFile argv arrays with a fixed project cwd (§5.2).
  • Validate branch names and file paths before passing to git.
  • Commit uses the repo's own configured git user (the operator's git identity) — do not hardcode an author.
  • Push: current branch only; set upstream on first push; never --force.
  • Merge-to-main: abort and surface a typed error on conflict — never leave a half-merged tree.
  • Confirm dialog before push and before merge-to-main (existing useConfirm).

Must Not

  • No new backend dependencies — git via execFile only (no simple-git/isomorphic-git).
  • No new secret storage — reuse the OS git credential helper.
  • No force-push, no history rewrite, no git reset --hard on user data.
  • Do not touch the DevEasy repo — operate strictly in the selected project's path.
  • No inline db()/SQL outside models (§7.4) — project path comes from ProjectModel as today.

Out Of Scope

  • Azure PR completion (PRS tab, plan 1 / existing Azure code). "Merge to main" here is local merge + push, per decision.
  • Interactive rebase, cherry-pick, stash management, conflict-resolution UI (conflicts are surfaced, resolution happens in the FILES editor / terminal).
  • Per-hunk staging (stage at file granularity in v1).

Risk

Level: 3 — Structural / security. Adds remote writes (push) and a near-irreversible op (merge to main) to a local app, plus command-execution surface.

Risks Identified

  • Command injection via branch name / commit message / path. Mitigation: argv arrays only (§5.2); branch-name allow-list; path guard; message passed as a single -m argv.
  • Merge conflict leaves the tree half-merged. Mitigation: detect non-zero merge, run git merge --abort, return GIT_MERGE_CONFLICT with the conflicting files; UI tells the user to resolve in FILES/terminal.
  • Push fails (auth, rejected non-fast-forward). Mitigation: surface GIT_PUSH_FAILED with a generic message (§3.4); log stderr internally; never auto-force.
  • Wrong-branch merge / accidental main push. Mitigation: explicit confirm dialogs naming the source and target branch; show current branch prominently.
  • Detached HEAD / unborn branch / no remote. Mitigation: guard each op (reuse the existing unborn-branch handling in getCurrentBranch); disable push/merge when no remote/upstream is configured and say why.

Blast radius: GitService, GitController, routes/git.ts, GitError, api/git.ts, useGit.ts, RepoTab.tsx + new repo/ children. Acts on real user repos on disk — the on-disk effect is the blast radius.

Pushback

  • Local merge to main bypasses review. You chose local-merge-+-push over Azure PR completion. For a protected main the push will be rejected anyway; for an unprotected main this lets you merge unreviewed. Recommended alternative if that becomes a problem: switch "merge to main" to Azure PR completion (the PRS tab already has the Azure client). Flagged, not blocking — proceeding with local merge per decision.

Method / PathBodyGitService op

POST /api/git/commit{ projectId, message }git commit -m <message> (staged)
POST /api/git/branch{ projectId, name }git checkout -b <name>
POST /api/git/push{ projectId }git push (set upstream first push)
POST /api/git/merge-main{ projectId }checkout main → merge branch → push; abort on conflict

Existing history, status, branches, checkout are unchanged. Each write returns the fresh GitStatus so the UI re-renders from one source.
