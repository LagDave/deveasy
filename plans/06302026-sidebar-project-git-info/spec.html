Why

The "Sessions by project" sidebar shows only the project name. Each project is a git repo and DevEasy already exposes working-tree status, but it is unused here. Surfacing the branch and change counts lets the operator see at a glance where each repo stands before opening a session.

What

Under each project name in the sidebar, show a compact line: the current branch (truncated) and counts of new, modified, and deleted files. Done when each project group renders its branch plus accurate change counts (or "clean"), driven by the existing git-status query.

Context

Relevant Files

  • frontend/src/components/Session/SessionSidebar.tsx — renders the project header row (only file changed)
  • frontend/src/hooks/queries/useGit.ts — useGitStatus(projectId), reused as-is
  • frontend/src/api/git.ts — GitStatus / GitStatusFile types (porcelain state codes)
  • src/controllers/git/feature-services/GitService.ts — backend source of the status data (unchanged)

Patterns To Follow

  • Data-fetching stays in the hook, not the component body (§14.3 / §15.1) — call useGitStatus from a per-project child component.
  • Icons via the shared Icon set (name="branch"); colors via theme tokens (text-success/text-accent/text-danger/text-faint).

Reference file: SessionSidebar.tsx — the existing RuntimeDot sub-component is the analog for a small presentational child.

Constraints

Must

  • Reuse the existing useGitStatus hook and GitStatus contract.
  • Truncate both the project name and the branch so the row never overflows the 18rem rail.
  • Bucket each file once: new (??/A) → deleted (D) → modified (else).

Must Not

  • No new dependencies, API endpoints, or backend changes.
  • No polling added — the status query fetches once and caches.
  • Do not refactor unrelated parts of the sidebar.

Out Of Scope

  • Ahead/behind indicators, per-file lists, click-through to a diff view.

Risk

Level: 1

Risks Identified

  • One git-status query per project = one git subprocess each. Mitigation: queries are cached with no refetch interval, so cost is one call per project per load; fine at the local single-operator scale.
  • Non-repo or errored projects. Mitigation: component renders nothing until data resolves, so a failed/absent status simply shows no git line.

Blast radius: SessionSidebar.tsx only; no consumers of the changed component signature (props unchanged).

Pushback

  • If the project count grows large, prefer a single batched status endpoint over N per-project queries. Not worth it now.
