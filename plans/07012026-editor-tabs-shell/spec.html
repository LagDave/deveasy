Why

Cockpit is a separate top-level tab, detached from the project you are actually editing. Everything project-scoped (files, git, pull requests, and soon a terminal) should live in one place: the editor takeover. This spec turns that takeover into a tabbed shell and removes the Cockpit sidebar entry, relocating its Azure pull-request panels into a PRS sub-tab.

This is the structural skeleton the other two plans build on: plans/07012026-repo-git-desktop fills the REPO tab, plans/07012026-terminal fills the TERMINAL tab.

What

The editor takeover (CodeEditorView) gains a tab strip with four tabs: FILES · REPO · PRS · TERMINAL. FILES holds today's file tree + editor unchanged. PRS holds the Azure PR panels scoped to the open project (no project selector — the project is already known). REPO and TERMINAL ship as empty placeholders in this plan (filled by plans 2 and 3); REPO temporarily hosts the existing read-only GitPanel so no git visibility is lost between plans.

The Cockpit entry is removed from the left sidebar. Done when: opening a project shows the four-tab takeover, FILES works exactly as before, PRS shows that project's PRs and connect/settings flow, and there is no Cockpit tab in the sidebar.

Patterns To Follow

  • §12.3 feature folders — new tab components live under components/CodeEditor/ (or a new components/Workspace/), not a flat dump.
  • §13.1 / §13.3 — CodeEditorView must not absorb every tab's body; each tab is its own child component (FilesTab, RepoTab placeholder, PrsTab, TerminalTab placeholder). One responsibility per file; stay well under the 800-line ceiling.
  • §15.1 / §15.2 — server state stays in React Query (existing Azure/git hooks); only the active-tab id is local UI state.
  • §15.1 — the PRS tab reuses the existing useAzure* / useGit* hooks; do not refetch or mirror into useState.

Reference file: frontend/src/components/CodeEditor/EditorTabs.tsx — match its tab-strip structure, active styling, and props shape for the workspace tab strip.

Constraints

Must

  • Keep FILES behavior byte-for-byte: tree, open tabs, dirty state, persisted editor state.
  • Scope PRS to the takeover's projectId — drop the Cockpit project selector.
  • Default active tab = FILES.
  • Preserve PR connect / reconnect (401 → settings) behavior from CockpitPanel.

Must Not

  • No new dependencies.
  • Do not build REPO git-write or TERMINAL logic here — placeholders only (REPO temporarily shows read-only GitPanel).
  • Do not change any backend route, controller, or model.
  • No drive-by refactor of the Azure PR components beyond removing the selector.

Out Of Scope

  • REPO commit/branch/push/merge and the diff viewer (plan 2).
  • Terminal / node-pty / xterm.js (plan 3).
  • Persisting the active tab across reloads (acceptable to default to FILES).

Risk

Level: 2 — Concern. UI restructure plus deleting a top-level entry; no backend change.

Risks Identified

  • Removing Cockpit before PRS exists would strip PR access. Mitigation: build the PRS tab in the same pass that removes the Cockpit entry; do not merge a state where neither exists.
  • Read-only git visibility (branch/status/history) lived only in Cockpit's GitPanel. Mitigation: REPO placeholder hosts GitPanel until plan 2 replaces it.
  • CockpitPanel.tsx becomes dead after the move. Mitigation: delete it in this plan (its children are reused; the container is not) to avoid a dead export (§ post-exec Step 1).
  • The PR panels assumed a nullable selected project; the takeover always has one. Mitigation: pass the non-null projectId through; remove the null-state branches that no longer apply.

Blast radius: App.tsx (sidebar), SessionPanel.tsx (unchanged mount), the five Cockpit child components (re-parented), CodeEditorView.tsx (wrapped). No backend consumers.

Pushback

  • Four tabs in a takeover that you reach per project means PRs are no longer reachable without opening a project. That is the intended consolidation, but if you ever want an all-projects PR overview, this removes it. Recommended: accept it now; revisit only if a cross-project view is actually needed.
