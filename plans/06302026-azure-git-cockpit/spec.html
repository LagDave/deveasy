Why

The team needs one place to see a project's git state and its Azure pull requests — history, open PRs, their status and review threads — and to open a new PR without leaving the app. This is the collaboration surface: a shared view of what's in flight. Scoped to Azure Repos via a per-user PAT for v1; pushing/merging stays manual for now.

What

A cockpit for the active project: local git history/branch/status read via the git CLI; an Azure DevOps integration (PAT auth) that lists pull requests, shows a PR's detail (status, reviewers/votes, comment threads, changed files), and creates a new PR. The PAT is stored in the OS keychain, never in the DB or repo. Read + create only — no merge, approve, or vote from the app in v1. Done when: with a PAT configured and an Azure repo mapped to the project, the user sees git history and live PRs, opens a PR detail, and creates a PR that appears in Azure.

Patterns To Follow

  • External HTTP (Azure) lives in a lib/ client wrapped by a feature-service; only DB access goes in models/ (§6.2, §7.4). The Azure client is not a model.
  • Backend domain shape mirrors gbp-automation/ (§6.1); envelope + typed error; correct status codes incl. 401 when the PAT is missing/expired (§8.3/§8.4).
  • Frontend triad (§12.1); PR/list data is React Query server state (§15.1).

Reference analogs: Constitution §6.1, §12.1. Builds on Spec 1's active-project + shell.

Constraints

Must

  • §5.1/§5.3 PAT stored only in the OS keychain; never written to the DB, repo, logs, or the frontend bundle
  • §5.2 sanitize project paths before invoking git; run git with an explicit cwd, never shell-interpolate user input
  • §7.4 only the mapping table touches the DB via a model; Azure data is fetched live (optionally cached later)
  • §8.4 map a missing/expired PAT and Azure auth failures to 401 with a clear "reconnect" code
  • §9 Pino with project context; redact the PAT from all logs

Must Not

  • No merge / approve / vote / push actions in v1 (read + create PR only)
  • Do not post AI review output back to Azure (in-app panel only — that is a later slice)
  • Do not store Azure PR bodies as the source of truth — fetch live
  • No wildcard CORS; the API stays bound to localhost (§11.4)

Out Of Scope

  • GitHub/GitLab providers (Azure only for v1; pluggable provider interface is a nice-to-have, not required)
  • Automated push, merge, branch policies, build/pipeline views
  • OAuth / Azure AD app (per-user PAT only)

Risk

Level: 3 — secret handling and an external API with auth.

Risks Identified

  • PAT leakage. A PAT in the DB, a log line, or an error response is a credential leak. Mitigation: keychain-only storage via SecretService; a redaction helper applied before logging; generic external error messages (§3.4); the PAT never crosses to the frontend (only a "connected/expired" status does).
  • PAT expiry mid-use. Azure PATs expire. Mitigation: detect 401/403 from Azure, surface a typed AZURE_RECONNECT_REQUIRED → HTTP 401, and a UI affordance to re-enter the PAT.
  • Command injection via git. Mitigation: invoke git via execFile/spawn with an argv array and a fixed cwd — never a shell string with interpolated input (§5.2).
  • Azure REST shape assumptions. API versions differ. Mitigation: pin the api-version query param in azureClient; type the responses; fail loud on unexpected shapes.

Blast radius: one new mapping table + keychain entries + outbound calls to Azure. Reads the active project from Spec 1; independent of Spec 2 and Spec 4.

Pushback

  • PATs are user-managed and expire — recurring friction. Accepted for v1 per decision; Azure AD OAuth is the future upgrade. Flagged, not blocking.

Rev 3 - 2026-06-30

Change: Decoupled the Cockpit from the now-removed global "active project". It has its own project dropdown (from the project list); git and Azure endpoints take an explicit projectId (query param for GETs, body for connect/create-PR), validated server-side and resolved via ProjectModel.findById. No more getActiveProjectId usage.

Reason: The "active project" concept was removed app-wide (app-shell Rev 3); the Cockpit needed an explicit project source.

Pushback (recorded): these endpoints trust the client-supplied projectId with no ownership check — consistent with the v1 single-operator/no-auth design, but if DevEasy goes multi-user, projectId becomes untrusted input needing server-side tenant scoping (§5.5/§11.7).
