Why

DevEasy wraps the Claude CLI but exposes none of its live controls. The operator cannot switch models, discover or invoke skills, compact a long chat, or see how much of their subscription limits or context window a session has consumed. Every input for these already flows through the pipeline (the CLI init and result stream events) or is reachable over a verified endpoint (the OAuth usage API) — it is simply never surfaced. This closes that gap so the wrapper is a real cockpit, not a blind pipe.

What

Five capabilities, all driven from existing data sources:

  • In-chat model switching — a picker that restarts the session's CLI process with --resume <cli-session-id> --model <alias> so the conversation context carries over; the resolved model (from the init event) is shown and updates live.
  • Skill autocomplete — typing / in the composer suggests the session's actual skills (from the init event's slash_commands/skills); a /-prefixed message bypasses the workflow-command gate and sends as-is.
  • /compact control — a button that sends /compact to the session to compress its context.
  • Subscription usage ring — a circular multi-track ring at the top of the sidebar showing 5-hour and weekly utilization; clicking opens a dropdown with each window as a horizontal bar plus its reset time. Data from GET https://api.anthropic.com/api/oauth/usage.
  • Context-window ring — a circular ring to the right of the Send button showing percent of the model's context window used by the current session, derived from result/init token counts.

Done when: the operator can change model mid-chat and keep context, autocomplete and send a skill, compact a chat, and read both usage rings with live values — with the backend never leaking the OAuth token and never crashing if the usage endpoint is unavailable.

Context

Verified facts (this is what the design rests on)

  • The CLI system/init stream event carries model (e.g. "claude-opus-4-8[1m]"), session_id, slash_commands, and skills. The result event carries usage (input/output/cache tokens), modelUsage (with contextWindow), and session_id. Both already flow through SessionStreamService.onProcessEvent and are persisted — nothing extracts them yet.
  • The CLI has no models list command. --model accepts stable aliases (opus, sonnet, haiku, fable) or full names. Aliases always resolve to the latest of a tier; the resolved version comes back in init.model. The model picker therefore uses a curated alias list, not a live query.
  • The CLI supports --session-id <uuid> (assign our own id at start) and --resume <id> — together these let a model switch restart the process while preserving conversation context.
  • GET https://api.anthropic.com/api/oauth/usage returns { five_hour: { utilization, resets_at }, seven_day: { utilization, resets_at }, limits: [...] }. Verified live: HTTP 200. Auth = the OAuth access token from the macOS Keychain entry "Claude Code-credentials", plus header anthropic-beta: oauth-2025-04-20. Undocumented and rate-limits aggressively (GitHub anthropics/claude-code #31637) — must cache and poll slowly.

Relevant Files

  • src/services/ClaudeProcessService.ts — spawns the CLI with a hardcoded arg set; one process per session. Gains a model arg, an own --session-id, and a resume-restart path.
  • src/controllers/sessions/feature-services/SessionStreamService.ts — owns session runtime, relays/persists events. Captures init (model, skills, cli session id), handles a set-model command, relays capabilities + context usage.
  • src/models/SessionModel.ts — all session DB access (§7.4). Gains model + cli_session_id read/write.
  • src/database/migrations/0002_sessions.ts — the table this migration extends (analog for the new migration).
  • src/routes/sessions.ts, src/controllers/sessions/SessionsController.ts — thin route + controller pattern to mirror for the usage route/controller.
  • frontend/src/components/Session/SessionComposer.tsx — command picker + textarea; gains model picker, skill autocomplete, / gate-bypass, /compact + context ring.
  • frontend/src/hooks/useSession.ts — the WS hook; gains model/context/skills state and a setModel action.
  • frontend/src/api/sessions.ts, frontend/src/components/Session/SessionSidebar.tsx (or the panel that renders the project-grouped list) — the sidebar host for the usage ring.

Patterns To Follow

  • Backend domain shape (§6.1, §6.3): mirror controllers/sessions/ — thin controller → feature-services/ business logic → models/ DB. The new usage read is a system integration, so the keychain+HTTP logic is a top-level shared service (src/services/ClaudeUsageService.ts), like ClaudeProcessService.ts; a thin UsageController + route exposes it.
  • Response contract (§8.1–§8.3): reuse controllerResponses ok()/handleSessionError() shape and a typed error (mirror ClaudeProcessError/SessionError).
  • Logging (§9.1): Pino childLogger only — never console.*. Never log the token (§5.3).
  • Frontend triad (§12.1, §14.2, §15.1): api/usage.ts thin typed functions over api/index.ts → a React Query hook useUsageLimits() → presentational ring components. No raw fetch in components.
  • Migration (§10.3): mirror 0002_sessions.ts — a single TypeScript Knex migration 0005_session_model.ts for Postgres. Deviation from the CLAUDE.md three-dialect migrations template: DevEasy is Postgres-only with TS migrations, so no mssql.sql. The plan's migrations/ folder carries the TS scaffold and a Postgres reference DDL only.

Reference files: backend — src/services/ClaudeProcessService.ts + src/controllers/sessions/SessionsController.ts; frontend — frontend/src/components/Session/SessionComposer.tsx + frontend/src/hooks/useSession.ts + frontend/src/api/sessions.ts.

Constraints

Must

  • Keep the OAuth token server-side only — read in ClaudeUsageService, never sent to the frontend, never logged (§5.1, §5.3, §17.3). The frontend receives only computed utilization/reset values.
  • Cache the usage-endpoint response in-memory with a TTL (≥60s) and back off on HTTP 429; degrade gracefully (ring renders "—") if the endpoint fails — the app must never break because Anthropic changed an undocumented endpoint (§3.1, §3.2).
  • Preserve conversation context on model switch via --resume with a stable --session-id set at first start; only allow switching when the session is idle, never mid-turn.
  • All session DB access stays in SessionModel (§7.4); the new column added via a migration (§10.3).
  • Continue scrubbing ANTHROPIC_API_KEY from the child env — subscription auth is load-bearing and unchanged.
  • Keep stream-event handling permissive — unknown event shapes still flow through untouched.

Must Not

  • No new runtime dependency for the rings — draw with inline SVG/CSS, not a charting library (§4.4).
  • Do not poll the usage endpoint on render or per-keystroke — it 429s aggressively.
  • Do not store the OAuth token in the DB or any DevEasy file.
  • Do not change the subscription-auth model or the one-process-per-session model.
  • No drive-by refactors of unrelated session code.

Out Of Scope

  • A full historical usage dashboard / per-project rollup (the chosen scope is the sidebar limit ring + the context ring only).
  • Persisting captured skills lists to the DB — they are per-session and relayed live; no storage needed.
  • Showing real dollar cost as a headline (subscription has no per-token bill; if any $ shown elsewhere later it is labeled "est. API-equivalent").
  • Auth/multi-user — DevEasy stays single-operator local-first.

Risk

Level: 3 — three structural unknowns; all isolated and mitigated below.

Risks Identified

  • (R-A, L3) Undocumented usage endpoint. /api/oauth/usage is community-discovered and rate-limits hard. Mitigation: isolate in one service behind /api/usage/limits; cache ≥60s; backoff on 429; graceful "—" on any failure. If Anthropic removes it, only the ring degrades.
  • (R-B, L3) OAuth token handling. A real credential is read from the Keychain. Mitigation: token never leaves the backend service, never logged, never persisted; only derived numbers cross the API boundary (§5.3).
  • (R-C, L3) CLI slash commands over stream-json stdin. /compact and /skill-name are interactive-mode constructs; their behavior when written to -p --input-format stream-json stdin is unverified. Mitigation: execution must confirm with a live probe before wiring the UI; if stdin slash does not trigger them, fall back (document the limitation, keep the discovery UI, and for compact surface a clear "not supported in this mode" rather than a silent no-op). This is the single biggest execution unknown.
  • (R-D, L3) Model-switch resume correctness. Kill + --resume --model may drop context or reject. Mitigation: assign our own --session-id at first start; verify resume replays context; block switching unless idle; on resume failure, surface an error and leave the old process intact where possible.
  • (R-E, L2) Restart race. Switching model while streaming. Mitigation: disable the picker unless runtime state is idle.
  • (R-F, L2) Schema drift in usage payload. Field names could change. Mitigation: parse defensively, prefer the structured limits[] array with five_hour/seven_day as fallback.

Blast radius: ClaudeProcessService.start() signature changes → consumers: SessionStreamService.attach() (only caller). SessionModel.create() signature → consumer: SessionsController.create(). useSession() return shape grows → consumers: SessionComposer + the session panel. WS message protocol gains set_model / capabilities / model_updated / context event types → consumer: useSession. New surfaces (usage service/route/api/hook/rings) have no existing consumers.

Pushback

  • The usage ring depends on an undocumented endpoint — accept it, but quarantine it. It is the right data and it works today, but it is not a contract. Building it behind one service with hard graceful-degradation is non-negotiable; the rest of the app must be indifferent to its failure. Recommended path taken in T5.
  • Verify R-C before building the composer UI. If /compact and /-skills do not work over stream-json stdin, the autocomplete is still useful for discovery but the "send" behavior changes. Do the probe first (T2 acceptance) so we are not wiring UI to a dead path.
