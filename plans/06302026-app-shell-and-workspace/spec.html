Why

DevEasy is a local-first web app that wraps the Claude Code CLI so a team shares one version-controlled set of agents, skills, and CLAUDE.md. This spec is the foundation every other slice builds on: a running Express + React app, a Docker Postgres for app state, a project picker over a gitignored projects/ folder, and the mechanism that injects the shared agent config into whichever project you select. Nothing else (sessions, git/Azure, agent editor) can run until this exists.

What

A working app: backend Express API on :1234 serving the built React/Vite SPA, a Knex/PostgreSQL connection to a Docker Postgres on :1231, and config validated at startup. The UI lists every clone in projects/, lets you select one as the active project, and on selection the backend injects the shared CLAUDE.md, .claude/skills, and .claude/agents into that project via symlink and adds them to the project's .gitignore. Done when: the app boots, the picker lists real folders, and selecting one produces verified symlinks + a managed gitignore block in the target project, with the active project persisted in Postgres.

Constraints

  • §6.2 every file in its correct top-level folder; §6.3 in-domain code under feature-services//feature-utils/
  • §7.1/§7.4 Routes → Controllers → Services → Models; all DB access in models/
  • §8.1 the two-shape response envelope on every endpoint
  • §5.6 validate env/config (ports, projects/ path, DB URL) at startup, fail fast
  • §5.2 path-traversal guard on every project path — resolve and assert it stays inside projects/
  • §9.1 Pino logger, no console.*
  • §12.1/§14.2 all frontend HTTP through api/index.ts

Must Not

  • No commit of projects/ into the DevEasy repo — it is gitignored
  • Do not overwrite a project's existing CLAUDE.md/.claude silently — detect and warn (graceful degradation)
  • No new state library on the frontend (§15.4)
  • No business logic in routes (§7.2)

Out Of Scope

  • Claude session spawning (Spec 2), git/Azure (Spec 3), agent editor (Spec 4)
  • Auth/login — local-first single-user-per-instance; no auth layer in v1

Risk

Level: 3 — the injection step writes symlinks and a .gitignore block into the user's real project repos.

Risks Identified

  • Clobbering a project's own config. A project may already have CLAUDE.md or .claude/. Mitigation: before injecting, detect existing non-symlink files; if found, refuse and surface a warning rather than overwrite. Only manage symlinks DevEasy created (tagged via a managed gitignore block with sentinel comments).
  • Path traversal. A crafted project name could escape projects/. Mitigation: resolve the absolute path and assert it is a direct child of the configured projects/ root before any FS op (§5.2).
  • Symlink portability. Symlinks behave differently if the repo is later opened on Windows. Mitigation: document macOS-first; the gitignore entries mean the links never enter version control regardless.
  • Repo-root config boundary (verified). Because each project is its own git repo, Claude's config walk stops at the project root and will not see the DevEasy root config — injection is therefore mandatory, not optional. This is the core design constraint, not a risk to avoid.

Blast radius: the FS injection touches files inside the selected project only. New consumers: Specs 2–4 all read the active-project state this spec persists.

Pushback

  • Copying config instead of symlinking would be safer against symlink edge cases but breaks live sync (edits to shared config wouldn't propagate). Recommended: symlink, with the existing-file guard above. Revisit if Windows support lands.
